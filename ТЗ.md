# Юрайт: Дела — CRM для юристов

## Техническое задание · v1.2 (Актуализированное под Python/FastAPI) · Май 2026

### 1. Контекст и цель проекта

**Юрайт·Суды** — публичный сервис поиска судов и дел. Пользователи находят свои дела через публичный поиск и хотят вести их дальше: отслеживать обновления, ставить задачи, хранить заметки и документы.

Для этого создаётся модуль **«Юрайт: Дела»** — закрытая CRM для юристов, доступная по подписке.

* 
**Ключевая идея:** карточка дела в CRM визуально перекликается с публичной карточкой на Юрайт·Суды. Пользователь находит дело на паблике → переходит в CRM → продолжает работу в знакомом интерфейсе.


* Также возможен переход в приложение с других URL основного сайта и иных маркетинговых каналов.

### 2. Основные параметры и Технический стек

#### Основные параметры:

* 
**Тип приложения:** Web-приложение (десктоп + адаптивный мобильный PWA).


* 
**Пользователи:** 1–2 юриста на аккаунт (однопользовательская модель на MVP).


* 
**Авторизация:** Строго Keycloak (`id.you-right.ru`, realm `You-right`).


* 
**Данные дел:** REST API Юрайт·Суды (`api.you-right.ru`).


* 
**Архитектура деплоя:** Полное контейнерное окружение (**Docker / Docker Compose**), оркестрация на VPS, автоматический деплой через **GitHub Actions**.



#### Выбранный стек технологий:

* **Бэкенд:** Python 3.11+ / FastAPI / Uvicorn.
* **База данных:** PostgreSQL.
* **ORM:** SQLAlchemy 2.0+ / Alembic (для миграций).
* **Фронтенд:** Next.js 14 (App Router) + TypeScript + Tailwind CSS (клиентская сборка без Node.js бэкенда, либо Next.js выступает исключительно как легкий BFF / прокси для статики).
* 
**Авторизация:** Проверка Keycloak JWT на стороне FastAPI (Dependency/Middleware), на фронтенде — NextAuth.js (Keycloak Provider).


* 
**Документы:** Внешние ссылки на папки Яндекс.Диска.

### 3. Авторизация (Keycloak) и интеграция с БД

Keycloak развёрнут на `id.you-right.ru`, realm `You-right`. Публичный сайт использует `client_id=allcourts`. Для CRM создаётся новый клиент:

* 
**Client ID:** `yurayt-dela` 


* 
**Client authentication:** ON (confidential client) 


* 
**Valid redirect URIs:** `https://cases.you-right.ru/api/auth/callback/keycloak` 


* 
**Страница входа:** Ванильная (стандартная от Keycloak) на MVP.



#### Логика синхронизации пользователя:

1. Фронтенд / NextAuth получает от Keycloak JWT-токен.


2. При каждом запросе к FastAPI в заголовке `Authorization: Bearer <token>` передается этот JWT.
3. Бэкенд на FastAPI валидирует токен через JWKS Keycloak и извлекает поле `sub` (уникальный ID пользователя).


4. Выполняется операция **upsert** в локальной таблице `users` по полю `keycloak_sub`. При первом входе запись создаётся автоматически, лимиты по умолчанию выставляются на тариф `free`.


5. Локальный `user_id` используется во всех внутренних связях БД (дела, задачи, заметки).

### 4. Модель данных и Биллинг

#### 4.1 Модель данных (Таблица `users` в PostgreSQL)

В таблицу пользователей добавляются следующие поля подписки:

| Поле | Тип | Описание |
| --- | --- | --- |
| `plan` | enum | Тариф: `free`, `standart`, `pro`, `custom` 

 |
| `cases_limit` | integer | Максимум дел на мониторинге (5 / 30 / 100) 

 |
| `subscription_ends_at` | datetime | Дата окончания подписки. `null` = бессрочно 

 |
| `bonus_balance` | decimal | Бонусный баланс в рублях (начисляется внешним хабом) 

 |
| `billing_updated_at` | datetime | Время последнего обновления лимитов 

 |

#### 4.2 Взаимодействие с платёжным хабом (Вебхук)

Платёжный хаб Юрайт после оплаты дёргает вебхук на бэкенде CRM:

* 
**Эндпоинт:** `POST /api/billing/update` 


* 
**Авторизация:** Bearer токен (`BILLING_WEBHOOK_SECRET` в `.env`) 


* 
**Тело запроса (JSON):** `{ "keycloakSub": "str", "plan": "str", "casesLimit": int, "subscriptionEndsAt": "ISO-string", "bonusBalance": float }` 


* 
**Действие CRM:** Поиск юзера по `keycloak_sub` → обновление полей подписки → ответ `200 OK`.



#### 4.3 Применение лимитов на стороне FastAPI

Бэкенд жестко валидирует лимиты в следующих точках:

1. 
`POST /api/cases/save`: Если количество активных отслеживаемых дел (собранных через парсер) $\ge$ `cases_limit`, возвращается **ошибка 402 (Payment Required)**. Фронтенд отображает экран «Лимит исчерпан».


2. 
`GET /api/cases/` и любые эндпоинты мутации: Если `subscription_ends_at < now()`, CRM переходит в режим **Read-only**. Фронтенд блокирует создание/редактирование и выводит красный баннер.



Примечание: Дела, которые просто находятся в общем кеше Юрайт·Суды и не поставлены пользователем на активный мониторинг, под лимиты не попадают.

4.4 Тарифная сетка MVP 

* 
**free:** 5 дел (Бесплатно при регистрации).


* 
**standart:** 30 дел (990 руб/мес).


* 
**pro:** 100 дел (2990 руб/мес) — основной тариф.


* 
**custom:** Любой лимит (Договорная цена).



#### 4.5 Ручное управление лимитами на MVP

Пока платёжный хаб не запущен, управление лимитами осуществляется через:

1. Прямое редактирование БД PostgreSQL (через pgAdmin / DBeaver).
2. Специальный CLI-скрипт на Python:


`docker compose exec backend python scripts/set_billing.py --email user@example.com --plan pro --limit 100 --days 30`.

### 5. Интеграция с API Юрайт·Суды и Фоновые задачи

#### 5.1 Получение данных о делах

Все данные запрашиваются из внешнего API `api.you-right.ru`:

* 
**КАД и ГАС (РФ):** `GET https://api.you-right.ru/kad/getCases` 


* 
**ГАС (Альтернативный):** `POST https://api.you-right.ru/gas/getCase?key={YOU_RIGHT_API_KEY}` 


* 
**Параметры запроса:** `case_number`, `key=YOU_RIGHT_API_KEY`, `priority=true|false`.


* 
`priority=true` (Запрос из интерфейса от пользователя): ускоренный импорт, если дела нет в кеше. Если статус ответа `"importing"`, фронтенд повторяет запрос каждые 4–5 секунд.


* 
`priority=false` (Фоновая синхронизация): обычная низкоприоритетная очередь.





#### 5.2 Фоновая синхронизация (Воркер)

Вместо Vercel Cron используется фоновый процесс на Python (например, **Celery Beat**, легковесный **Rocketry** или планировщик **APScheduler** внутри отдельного Docker-контейнера).

* 
**Расписание:** 2 раза в сутки (в 03:00 и 15:00 UTC).


* 
**Логика:** Выбираются только дела, у которых `is_watched == True` и заполнен `case_number`. Бэкенд делает запросы к API с параметром `priority=false`. При обнаружении новых событий создаются системные уведомления (`is_read = False`).

### 6. Экраны приложения (MVP)

Интерактивный HTML-прототип интерфейсов доступен в файле `index.html`.

1. 
**Главная (Dashboard):** 4 метрики (дела, заседания, обновления, задачи). Список последних 5 дел, мини-календарь недели, плашка подписки внизу сайдбара.


2. 
**Мои дела:** Фильтрация (Активные / Завершённые / Архив). Дела без номера выделяются серым цветом с бейджем «Подано».


3. 
**Карточка дела:** Номер, источник, кнопка «Отслеживать». Метрики (сумма иска, заседание, события). Табличные данные (суд, судья, стороны + ИНН). Сворачиваемые секции: история КАД, задачи, заметки, документы (ссылка на Яндекс.Диск).


4. 
**Мастер добавления дела (3 шага):** Шаг 1 — Поиск (КАД / ГАС / Без номера). Шаг 2 — Редактирование метаданных, привязка Яндекс.Диска. Шаг 3 — Фиксация.


5. 
**Задачи:** Общий список с дедлайнами (просроченные подсвечены красным). Привязка к делу опциональна.


6. 
**Заметки:** Двухпанельный режим (список слева, редактор с автосохранением справа).


7. 
**Уведомления:** Лента системных событий из судов. Клик по уведомлению переводит в карточку дела и снимает отметку «не прочитано».


8. 
**Календарь:** Календарная сетка заседаний на месяц.

### 9. Инфраструктура и CI/CD (GitHub Actions)

Проект полностью упаковывается в **Docker**-контейнеры. В корне находится `docker-compose.yml` для локальной разработки.

#### Конвейер GitHub Actions (`.github/workflows/deploy.yml`):

1. **Раздел тестирования (CI):** При каждом Pull Request в ветки `main` и `develop` запускается проверка линтеров (Flake8 / Black / Ruff для Python, ESLint для TypeScript) и прогон базовых тестов (Pytest).
2. **Раздел деплоя (CD):** При слиянии в ветку `main`:
* Сборка Docker-образов для `backend` и `frontend`.
* Публикация образов в Docker Registry (GitHub Packages или Docker Hub).
* Подключение по SSH к целевому VPS серверу.
* Обновление контейнеров через `docker compose pull && docker compose up -d`.
* Автоматический запуск миграций базы данных (`alembic upgrade head`).
